Operating Systems: Three Easy Pieces — Explained Simply

The classic OSTEP textbook by Remzi & Andrea Arpaci-Dusseau, broken down chapter by chapter. No jargon, no complexity analysis. Just the key ideas, explained like you're in high school.

Prelude
Introduction
What is an operating system, and why should you care?
Chapter 1

The Invisible Manager Behind Your Computer

How does a computer let you run a browser, music player, and game all at the same time with just one processor?

Programs think they own the hardware. The OS secretly shares it among everyone.
Analogy

The OS is like the manager of a busy restaurant. You never see them cooking or serving, but they coordinate everything so every customer (program) gets served. Through virtualization, each program thinks it has the entire computer to itself.

The OS has three big jobs: (1) virtualizing hardware so programs are easy to write, (2) handling concurrency when multiple things happen at once, and (3) storing data persistently so your files survive a reboot. It's the invisible referee, traffic cop, and librarian of your computer, all rolled into one.

Part I
CPU Virtualization
How one CPU pretends to be many, and who gets to run next.
Chapter 2

Bringing Programs to Life: The Process

A program on your hard drive is just a file. How does the OS turn it into a living, running thing?

Analogy

Think of a recipe (program) versus actually cooking a dish (process). The recipe on paper does nothing. But when a chef starts following it -- gathering ingredients, chopping, stirring -- that's a process. A process is a running program with its own memory, registers, and a current step it's on.

The three states of a process: Running, Ready, or Blocked.

The OS keeps a list of all processes, noting each one's state. When a process does something slow like reading a file, it becomes blocked and the OS hands the CPU to someone else -- no time wasted. The OS tracks each process's full state in a Process Control Block -- like a bookmark that lets it pause a process and perfectly resume it later.

Chapter 3

Fork, Exec, and Wait: The UNIX Family Recipe

How do you actually create and control processes in a real operating system?

Analogy

Creating a process in UNIX is like biological reproduction. fork() creates a child that's an almost identical clone of the parent -- like a cell dividing in two. Then exec() transforms the child into a completely different program, and wait() lets the parent pause until the child finishes.

When you type "ls": shell forks, child exec's into ls, parent waits.

This fork-then-exec design is brilliant because between the two steps, the shell can set up things like redirecting output to a file -- which is exactly how ls > output.txt works.

Chapter 4

Baby-Proofing the CPU

If you let a program run directly on the CPU (for speed), how do you stop it from doing dangerous things or hogging the processor forever?

Analogy

It's like baby-proofing a room. Before you let the toddler (program) roam free, you lock the cabinets (restrict dangerous operations) and cover the outlets (block direct hardware access). Then you set a kitchen timer (hardware timer interrupt) so you check back regularly.

Programs run fast in user mode, but must "trap" into kernel mode for dangerous operations.

A hardware timer fires every few milliseconds, interrupting whatever program is running and giving the OS a chance to decide: should I let this keep running, or switch to another? This is called a context switch, and it's how the OS stays in charge.

Chapter 5

Who Gets to Go Next? CPU Scheduling

When multiple programs want the CPU, which one should run first?

Analogy

It's like managing checkout lines at a grocery store. FIFO seems fair -- whoever arrives first goes first. But if someone with a full cart is ahead of you and you only have one item, you wait forever (the convoy effect). Shortest Job First is like an express lane. Round Robin gives everyone 30 seconds at the register before rotating.

Three jobs arrive: A takes 100s, B and C each take 10s. With FIFO (A first), average wait is 110s. With SJF (B and C first), it drops to 50s. But SJF is terrible for response time. Round Robin fixes response time but makes turnaround worse. You can't optimize for everything at once.

Chapter 6

The Smart Line That Watches You: MLFQ

The best schedulers need to know how long each job takes -- but the OS has no crystal ball. How do you schedule without knowing the future?

Analogy

Imagine a theme park with VIP, regular, and economy lines. Every new visitor starts in VIP. If they take a quick ride and leave, great. But if they keep riding for a long time, they get moved down. The system learns from your behavior instead of asking you upfront.

MLFQ: new jobs start at the top. CPU hogs get demoted. Interactive jobs stay high.

To prevent starvation, the system periodically boosts everyone back to the top. MLFQ powers Windows, macOS, and many Linux systems.

Chapter 7

The Lottery Ticket Approach: Proportional Share

Instead of fastest-first, what if you want to guarantee each process gets a fair percentage of the CPU?

Analogy

Imagine a raffle. You have 75 tickets and your friend has 25. Every second, a winning ticket is drawn and the winner gets the CPU. Over time, you win about 75% of the draws. Simple, random, and surprisingly fair.

Linux's Completely Fair Scheduler (CFS) takes a similar approach: it always runs the process that has received the least CPU time so far, using an efficient red-black tree to find it instantly, even with thousands of processes.

Chapter 8

Managing a Team of Chefs: Multi-CPU Scheduling

Modern computers have multiple CPU cores. How do you divide work across them without chaos?

Analogy

Imagine a restaurant with four chefs sharing one order board (single-queue). They crowd around it, and keep switching stations. A better idea: give each chef their own order board (multi-queue). The catch? Sometimes one chef is overwhelmed while another is idle -- so occasionally, a chef steals an order from a busy colleague.

Multi-queue scheduling keeps jobs on the same CPU for better cache affinity, with work stealing to balance load. There's no single perfect answer -- Linux has tried multiple approaches (O(1), CFS, BFS).

Part II
Memory Virtualization
How every program believes it has all the memory in the world.
Chapter 9

The Illusion of Your Own Memory Kingdom

How can multiple programs run at the same time without stepping on each other's memory?

Analogy

You and your friends all live in the same apartment building, but each of you believes you own the entire building. The OS creates this illusion using an address space -- each program thinks it has memory all to itself, starting from address 0.

Each program sees code at the top, heap growing down, stack growing up.

The OS pursues three goals: transparency (programs don't know memory is shared), efficiency (it shouldn't slow things down), and protection (one program can't mess with another's memory).

Chapter 10

Memory's Drive-Thru Menu

How do programmers ask for memory and give it back?

Analogy

The stack is like a cafeteria tray -- food appears automatically when you sit down and gets cleared when you leave. The heap is like ordering from a menu with malloc() -- you ask for what you want, and you must return the tray yourself with free(). Forget to return it? That's a memory leak.

Common mistakes: forgetting to free (leak), freeing twice (crash!), or using memory after freeing it (dangling pointer -- like eating from a tray someone already took away).

Chapter 11

The Secret Translator

How does the hardware convert fake addresses into real physical addresses?

Analogy

Every letter you mail has your own made-up address system. The post office (hardware) has a secret formula: physical address = base + virtual address. If you try to access an address beyond your bounds, you get shut down -- no snooping into someone else's mail.

If the base register is 32KB and your program accesses virtual address 1000, the hardware calculates: physical = 32KB + 1000 = 33768. The hardware does the fast per-access translation; the OS handles the big-picture bookkeeping.

Chapter 12

Divide and Conquer: Segmentation

With simple base-and-bounds, the big unused gap between heap and stack wastes real memory. How do we stop that waste?

Analogy

Instead of moving your entire apartment into storage as one giant block (including empty rooms), store just the rooms you actually use. Segmentation breaks the address space into logical chunks -- code, heap, and stack -- each with its own base-and-bounds.

The downside? Over time, physical memory gets chopped into odd-sized free chunks that are hard to reuse -- external fragmentation. This problem motivates the next big idea: paging.

Chapter 13

The Tetris Problem of Memory

When memory is divided into variable-sized chunks, how do you manage the leftover free pieces?

Analogy

Imagine playing Tetris, but you can never rotate the pieces. As blocks land and get removed, you end up with annoying little gaps. Even if you have enough total free space, no single gap might be big enough.

Allocators use strategies: best fit (tightest hole), first fit (first hole that works), and buddy allocation (splitting in powers of two so merging is easy). Each trades off speed, simplicity, and fragmentation. No perfect answer -- just smart compromises.

Chapter 14

Cookie-Cutter Memory: Paging

Segmentation causes fragmentation because chunks are different sizes. What if we cut everything into identical pieces?

Analogy

Imagine a parking garage where every spot is the exact same size. No matter what car (data) you need to park, just find any open spot. A page table is the map: "your page 0 is parked in spot 3, page 1 in spot 7."

The page table maps virtual pages to physical frames. Pages can be scattered anywhere.

Paging eliminates external fragmentation entirely. But page tables can get huge (4MB per process for a 32-bit system!). The next chapters tackle making paging faster and smaller.

Chapter 15

The Speed Dial for Memory: TLBs

Paging requires looking up the page table for every memory access. Way too slow. How do we speed this up?

Analogy

You could look up every number in a giant phone book (page table), but that takes forever. Instead, memorize your most-called numbers. The TLB is that speed dial -- a tiny, super-fast cache of recent translations. 90%+ of accesses are instant hits.

A program accessing an array: the first access to each page is a TLB miss (slow), but every subsequent access to the same page is a hit (fast). Programs with good spatial locality (orderly access patterns) run much faster.

Chapter 16

Shrinking the Phone Book: Multi-Level Page Tables

Page tables are too big -- 4MB per process, 400MB for 100 processes!

Analogy

You have a massive phone book, but most pages are blank. A multi-level page table is like a table of contents: you only print the sections that actually have entries. Empty sections? Just "nothing here" in the table of contents.

Modern x86 systems use four levels of page tables. The memory savings are dramatic -- you only allocate page table space for parts of the address space a program actually uses.

Chapter 17

When Memory Isn't Enough: Swapping

What happens when all your programs need more memory than your computer physically has?

Analogy

Your desk (RAM) is small but fast. The filing cabinet (hard drive) is huge but slow. When your desk fills up, you put some papers in the cabinet. When you need them again, you pull them out. This is swapping.

When a program accesses a page not in RAM, the hardware triggers a page fault. The OS loads the page from disk, updates the page table, and restarts the program. It never knows this happened -- just a brief delay.

Chapter 18

Who Gets Kicked Out? Page Replacement

When memory is full and you need a new page, which one should you remove?

Analogy

Your desk is full and you need space. Which paper goes to the filing cabinet? The one you won't need for the longest time! That's the optimal policy -- but it requires a crystal ball. In practice, use LRU: kick out whatever you haven't touched in the longest time.

Real systems approximate LRU with the clock algorithm: sweep through pages like a clock hand, evicting ones that haven't been recently accessed. Prefer evicting "clean" pages since "dirty" ones must be written to disk first.

Chapter 19

Putting It All Together: Complete VM Systems

We've learned all the pieces. How does a real OS combine them?

Analogy

You've learned about bricks, plumbing, wiring, and roofing separately. Now see how an architect puts it all together. This chapter examines VAX/VMS (1970s classic) and Linux (modern standard).

Both use clever tricks like demand zeroing (don't clear a page until someone reads it) and copy-on-write (let processes share pages until one writes). A complete VM system is a carefully engineered combination of page tables, TLBs, replacement policies, and OS structures working together.

Part III
Concurrency
What happens when multiple things try to happen at the same time.
Chapter 20

When Cooks Share a Kitchen: Threads

How can a single program do multiple things at the same time without everything going haywire?

Analogy

A thread is like having multiple cooks in the same kitchen. They share the same fridge and counters (memory), but each cook has their own hands and recipe step (registers, program counter). If two cooks add salt to the same pot without talking, you get a race condition.

Two threads both read 50, both write 51. One update is lost -- a race condition.

Shared data plus uncontrolled scheduling equals unpredictable results. We need special tools (locks, condition variables) to fix this.

Chapter 21

The Thread Instruction Manual

How do you actually create threads, wait for them, and protect shared data in real code?

Analogy

Like assembling IKEA furniture: pthread_create = open the box and start building. pthread_join = wait until your friend finishes their part. Locks = the "do not disturb" sign so only one person uses a room at a time.

Golden rules: keep it simple, always initialize your locks, check return codes, and never return a pointer to a local variable from a thread.

Chapter 22

Building the Bathroom Door: Locks

How do we actually build a lock that prevents two threads from messing with the same data?

Analogy

A simple "check if locked, then lock" fails because two people can both check at the same instant and both walk in. The solution: test-and-set -- it checks AND locks the door in one single, unbreakable motion. Nobody can sneak in between.

A spin lock has you jiggling the handle in a loop -- wasteful. Better: take a number at the deli counter, sit down, get called when it's your turn. Linux uses a hybrid: try briefly, then sit down and wait.

Chapter 23

Making Your Bookshelf Thread-Safe

You have data structures that work fine with one thread. How do you add locks for multiple threads?

Analogy

One big padlock on an entire library (coarse-grained) = correct but slow. A separate lock on each bookshelf (fine-grained) = multiple people browsing different shelves at the same time.

An "approximate counter" gives each CPU core its own local counter (like each cashier having a tip jar). Periodically, local counts merge into the global total. Start simple with one big lock -- only get fancy if you measure a performance problem.

Chapter 24

The Waiting Room: Condition Variables

Sometimes a thread needs to wait for something specific to happen before continuing. How do you wait efficiently?

Analogy

Instead of asking the receptionist "Is the doctor ready?" every five seconds (spin-waiting), sit in the waiting room and get called when it's your turn. Use while loops (not if) to recheck after waking -- someone might have snatched the appointment before you.

The classic producer/consumer: a chef makes sandwiches, a customer grabs them. Use TWO condition variables (buffer full, buffer empty) and always recheck with while.

Chapter 25

The Swiss Army Knife: Semaphores

Locks and condition variables are two separate tools. Is there one that does both?

Analogy

A semaphore is a bouncer with a clicker-counter. Start at 5 (five seats). Each person entering decreases by one (sem_wait). At zero, wait outside. When someone leaves, count goes up (sem_post). Set initial count to 1 = a lock. Set to 0 = a signal.

The Dining Philosophers problem: five philosophers, five forks, everyone grabs left fork simultaneously -- deadlock! Fix: make one philosopher reach for forks in opposite order, breaking the cycle.

Chapter 26

The Bug Museum: Common Concurrency Bugs

What are the actual bugs that real-world multi-threaded programs suffer from?

Analogy

Like car accidents, most are caused by a few common mistakes. The two big ones: atomicity violations (you assumed steps would happen together, but another thread snuck in) and order violations (you assumed Thread A runs before B, but it didn't).

Thread 1 checks if a pointer is not NULL, then uses it. Between the check and use, Thread 2 sets it to NULL. Crash! Fix: wrap both inside a lock. For deadlock, the simplest fix is to always grab locks in the same order everywhere.

Chapter 27

One Juggler, Many Balls: Event-based Concurrency

Threads are powerful but tricky. Can we handle many tasks at once WITHOUT threads?

Analogy

Instead of hiring many waiters (threads) who bump into each other, have one super-efficient waiter who checks all tables in a loop: "Table 1 needs water? Done. Table 3 ordered? Send it." This is an event loop. One waiter = no locks needed! This is how Node.js works.

The catch: the waiter must NEVER get stuck at one table (blocking call). If something takes time, place the order asynchronously and move on. Neither threads nor events have "won" -- both remain useful.

Part IV
Persistence
How your data survives crashes, power failures, and the test of time.
Chapter 28

Hey Computer, Talk to Your Gadgets!

How does the OS communicate with devices (keyboards, disks, mice) without wasting the CPU's time?

Analogy

You order food at a restaurant. Polling = standing at the kitchen door staring until your meal is ready (wasteful!). Interrupts = sitting down and the waiter taps your shoulder when food arrives (smart!). DMA = a delivery robot brings food directly to your table without you or the waiter doing anything.

The OS uses device drivers -- translator programs that let it talk to any brand of keyboard, disk, or mouse using the same language, like a universal remote for different TVs.

Chapter 29

The Spinning Record Player of Data

How does a hard disk store and retrieve files, and why are some patterns so much slower?

Analogy

A hard disk is like a vinyl record player. Data is on spinning platters, and a tiny arm swings to find the right track (seek), then waits for the right spot to spin underneath (rotational delay). Reading songs in order = fast. Jumping between random tracks = painfully slow.

Disk schedulers sort requests like an elevator stopping at floors in order rather than randomly bouncing (the SCAN/Elevator algorithm).

Chapter 30

Teamwork Makes the Dream Work: RAID

A single disk is too slow, too small, or too risky. How can we combine multiple disks?

Analogy

Group project! RAID-0 splits work evenly -- super fast, but if one person is sick, everything's ruined. RAID-1 = everyone keeps a backup copy -- safe, but double the people needed. RAID-5 = the team stores a cheat sheet so if one member's work is lost, the others can reconstruct it.

Three RAID levels: speed (striping), safety (mirroring), or balance (parity).

The whole system looks like one big disk to your computer. RAID-5 uses XOR math so if one disk dies, the rest can reconstruct it.

Chapter 31

The Filing Cabinet of Your Computer

How does the OS let you organize, name, read, write, and share data on disk?

Analogy

Your file system is a giant filing cabinet. Each drawer is a directory. Inside are documents (files). Each has a secret ID (inode number) but you see friendly names like "homework.docx." When you open a file, you get a file descriptor -- a ticket number for continued access.

Hard links = two labels on the same folder. Symbolic links = sticky notes saying "go look over there." Deleting is really unlinking -- the data only disappears when no names point to it.

Chapter 32

Building a Filing Cabinet from Scratch

Given a blank disk, what data structures do you need to organize files?

Analogy

Organizing a warehouse: (1) a map of shelves showing which are empty (bitmaps), (2) a card catalog with item descriptions (inode table), (3) the actual shelves of stuff (data region), (4) a master info sheet at the entrance (superblock).

Opening "/foo/bar" requires ~6 disk reads (root inode, root data, foo inode, foo data, bar inode, bar data). That's why caching in RAM is critical.

Chapter 33

Keep Related Stuff Close: The Fast File System

The original UNIX file system scattered data everywhere, achieving only 2% of disk bandwidth. How can we fix this?

Analogy

A grocery store where milk, cereal, and bowls are in three different buildings across town. FFS redesigns the store so breakfast items are all in the same aisle (block group). Files in the same directory go in the same group.

For large files, FFS spreads chunks across multiple groups, but makes each chunk big enough that read time far outweighs the brief seek -- this is amortization.

Chapter 34

What If the Power Goes Out Mid-Save? Journaling

When the computer crashes mid-write, the file system can end up broken. How do we prevent this?

Analogy

Remodeling your kitchen: tear out counter, install plumbing, put in new counter. If a tornado hits after step 1 but before step 3, disaster! Journaling is keeping a to-do list in a fireproof safe. After the tornado, check the safe and finish the job. The old approach (fsck) = searching the entire house to figure out what went wrong -- painfully slow.

Journaling: plan first, commit, then apply. Crash before commit? Ignore. After? Replay.

Most systems use metadata journaling (only logging inodes and bitmaps, not data) to cut write overhead in half.

Chapter 35

Never Erase, Just Keep Appending: LFS

Traditional file systems scatter small writes across the disk. Can we make all writes sequential?

Analogy

Instead of erasing old diary pages and rewriting, just keep writing on the next blank page -- always forward. LFS buffers all updates in memory, then writes them as one big sequential chunk. Since the disk arm never jumps, writes approach maximum speed.

The downside: old data piles up. A cleaner periodically reads segments with mostly dead data, copies live blocks, and frees the rest. Amazing write speed versus garbage collection overhead.

Chapter 36

The Tricky Silicon Speed Demon: SSDs

SSDs have no moving parts but weird rules -- you must erase before writing, and erasing wears it out.

Analogy

Flash memory is like a whiteboard: read anything instantly, but you can't overwrite one word -- you must erase the entire board first. Worse, each board can only be erased ~10,000 times. The Flash Translation Layer (FTL) is a librarian who writes updates on fresh boards and maintains a lookup table, ensuring equal wear on all boards (wear leveling).

SSDs crush hard drives in random read/write speed (100x+!) but cost about 10x more per gigabyte. Most modern FTLs use a log-structured design -- just like LFS!

Chapter 37

Trust, but Verify: Data Integrity

Disks sometimes silently corrupt your data without telling you. How do you detect this?

Analogy

Mailing a letter: include a secret code (checksum) summarizing the contents. The recipient recalculates the code. If they match, the letter is intact. If not, something went wrong in transit. Storage systems compute checksums for every block of data.

For misdirected writes (data written to the wrong address), the checksum also stores the block offset. For lost writes (disk claims it wrote but didn't), keep separate checksum summaries. Regular disk scrubbing catches problems before all backups go bad.

Part V
Security
Protecting your system from attackers, inside and out.
Chapter 38

The Bodyguard of Your Digital World

How do we protect the OS -- the master program -- from attackers who want to steal data or cause chaos?

Analogy

The OS is like the foundation of a house. Beautiful walls and roof mean nothing if the foundation is sand. If the OS is compromised, nothing above it is safe. Three big goals: keep secrets secret (confidentiality), prevent tampering (integrity), and keep things running (availability).

Timeless design principles: "keep it simple," "default to safe settings," "always check permissions," and give programs only the minimum permissions they need (least privilege).

Chapter 39

Proving You Are Who You Say You Are

Before granting access, the computer needs to verify identity. How?

Analogy

Getting into a concert: something you know (password/secret phrase), something you have (ticket/phone token), or something you are (fingerprint/face). The best security uses two or more: multi-factor authentication.

The system stores a scrambled version of your password (hash). It adds random salt so two people with the same password produce different hashes. That's why short passwords are dangerous -- attackers guess millions per second.

Chapter 40

The Bouncer at the Digital Door: Access Control

Once we know who you are, how does the OS decide what you're allowed to do?

Analogy

Two ways to run an exclusive club: Access Control Lists (ACLs) = a bouncer with a guest list on each door. Capabilities = handing out keys -- if you have the key, you're in. Most real systems (like Linux) use ACLs: each file says what owner, group, and everyone else can do.

Role-Based Access Control is like uniforms: wearing the chef's hat gets you into the kitchen, not the manager's office. Assign permissions to roles, not individuals.

Chapter 41

Writing in Secret Code: Cryptography

How do we protect data when it leaves the safety of our OS -- traveling the internet or sitting on a stolen laptop?

Analogy

Putting your diary into a locked safe before mailing it. Symmetric = a shared locker combination (same key to lock and unlock). Public key = a mailbox: anyone can drop a letter in (encrypt with public key), but only the owner can open it (decrypt with private key).

The #1 rule: all security depends on keeping the key secret, not on hiding how the lock works. Modern phones use full disk encryption so a stolen phone yields only scrambled nonsense.

Chapter 42

Staying Safe Across the Internet

When computers talk over a network anyone can eavesdrop on, how do we keep conversations private?

Analogy

Sending a sealed letter through a chain of strangers. Certificates = a notarized ID card. A trusted authority vouches "yes, this public key belongs to Amazon.com." TLS/SSL sets up an encrypted tunnel using Diffie-Hellman key exchange -- two strangers agree on a shared secret while everyone listens, because the math is impossibly hard to reverse.

When you visit "https," your browser checks the certificate, creates a fresh shared secret via Diffie-Hellman, and encrypts everything. This entire handshake happens in a fraction of a second. Never type a password on "http" without the "s"!

Key Takeaways

  • Virtualization: The OS creates illusions -- each program thinks it owns the CPU and all of memory, when really everything is shared
  • Processes: Programs come alive as processes. The OS switches between them so fast, they all seem to run at once
  • Scheduling: There's no perfect scheduler. Fast turnaround vs quick response is a fundamental trade-off. MLFQ learns from behavior
  • Virtual Memory: Paging + TLBs + multi-level page tables = the illusion of infinite, private memory for every program
  • Concurrency: Shared data + uncontrolled scheduling = bugs. Use locks, condition variables, or event loops to stay safe
  • Persistence: File systems organize data on disk. Journaling prevents corruption on crash. SSDs are fast but need wear leveling
  • Security: Authenticate, authorize, encrypt. Use least privilege. Never trust the network without TLS
