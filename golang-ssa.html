Go SSA: Understanding Static Single Assignment from Scratch

SSA is the intermediate representation that powers compilers and static analysis tools. This post builds it up step by step — from the core idea, to phi nodes, to walking the graph in Go code.

The Problem: Mutable Variables Are Hard to Analyze

Consider this code: 

x := 1
x = x + 2
x = x * 3
fmt.Println(x)

A compiler or analysis tool wants to answer: "What is the value of x at line 4?" But x has been assigned three times. You have to trace through every assignment to figure out which one matters. In a big program with loops and branches, this gets hard fast.

The SSA Idea: Assign Once, Name Everything

Static Single Assignment says: give every assignment its own unique name. Rewrite the code above as: 

t0 = 1
t1 = t0 + 2
t2 = t1 * 3
print(t2)

Now every name has exactly one definition. If you see t1, you instantly know it equals t0 + 2. No ambiguity. No tracing needed.

The Rule: In SSA, every name is assigned exactly once. Need a new result? You get a new name. You never reuse an old one.

Values, Not Variables

In normal Go, you think in terms of variables — named boxes that hold data and can be overwritten: 

x := 1       // x is a box, it holds 1
x = x + 2    // same box, now holds 3

In SSA, each computation produces a value, and that value never changes: 

t0 = 1         // t0 is a value, it IS 1, forever
t1 = t0 + 2    // t1 is a value, it IS 3, forever

You can think of each SSA value as an immutable box — once you put something in, it's sealed shut. Need a new result? Get a new box.

The key consequence: if someone tells you "look at box t1", you know exactly what's inside without tracing through the program. There's only one place it was ever defined.

Instructions vs Values: Two Roles, One Object

In Go's SSA, there are two core interfaces:

  • Instruction — something that does something (a computation, a branch, a store)
  • Value — something that produces a result (a named, typed, immutable result)

Many SSA objects are both. A BinOp (binary operation like a + b) is simultaneously an instruction (it performs addition) and a value (it produces a result you can reference). One Go struct, two roles.

But not everything is both:

Produces a result (Value) No result
Does something (Instruction) BinOp, Call, Phi Store, Return, Jump
Just exists Parameter, Const —

Store writes to memory but produces no result you can name. Parameter is a function argument — it's a value, but no computation created it; it was given to you.

Phi Nodes: Merging Branches

What happens when a value depends on which branch was taken? 

func foo(x int) int {
    if x > 0 {
        x = 1
    } else {
        x = 2
    }
    return x
}

After the if/else, which value does x have? It depends on the path. In SSA, we can't just say x — every value has one definition. This is where phi nodes come in. 

b0:                        // entry
    t0 = Parameter {x}
    t1 = t0 > 0
    if t1 goto b1 else b2

b1:                        // then branch
    jump b3

b2:                        // else branch
    jump b3

b3:                        // after the if/else
    t2 = phi [b1: 1, b2: 2]
    return t2

t2 is a phi node. It means: if we arrived from b1, the value is 1; if from b2, the value is 2.

Phi vs Ternary: A ternary (x > 0 ? 1 : 2) evaluates a condition. A phi doesn't check any condition — it just looks at which block control flow came from. The condition was already evaluated earlier. The phi just merges the results after the fact.

A phi node is both an instruction and a value. As an instruction, it picks from its inputs based on the incoming edge. As a value, it produces a result (t2) that other instructions can use.

Loops: Phi Nodes Get Interesting

func sum(n int) int {
    s := 0
    for i := 0; i < n; i++ {
        s = s + i
    }
    return s
}

In SSA: 

b0:                              // entry
    t0 = Parameter {n}
    jump b1

b1:                              // loop header
    t1 = phi [b0: 0, b2: t3]        // i: starts at 0, updated from b2
    t2 = phi [b0: 0, b2: t4]        // s: starts at 0, updated from b2
    t5 = t1 < t0                     // i < n
    if t5 goto b2 else b3

b2:                              // loop body
    t4 = t2 + t1                     // s + i
    t3 = t1 + 1                      // i + 1
    jump b1

b3:                              // after loop
    return t2

Notice something odd in b1: 

t1 = phi [b0: 0, b2: t3]

t3 is defined in b2, which comes after b1 in the listing. The phi references a value that appears later in the text.

This is fine. Don't read SSA like sequential code. The phi declares a rule: "on each arrival to this block, pick the right value." On the first visit (from b0), t1 is 0. On later visits (from b2), t1 is t3 — which was computed in the previous loop iteration. By the time control flow arrives from b2, t3 already exists.

This is how SSA handles mutable loop variables without any mutation.

Walking the SSA Graph

Every SSA value has a Referrers() method that returns all instructions that use it. Every instruction has an Operands() method that returns the values it depends on. These two methods give you the entire graph.

Forward: Definition → Uses

// grab the first parameter (n)
param := fn.Params[0]

// find everything that uses n
for _, instr := range *param.Referrers() {
    fmt.Printf("  used by: %s\n", instr)
}
// Output: used by: t1 = i < n

One call to Referrers() and you have every use. You can chain it to walk multiple steps: 

func walkUses(v ssa.Value, depth int) {
    indent := strings.Repeat("  ", depth)
    for _, instr := range *v.Referrers() {
        fmt.Printf("%s%s\n", indent, instr)
        // if this instruction also produces a value, keep walking
        if next, ok := instr.(ssa.Value); ok {
            walkUses(next, depth+1)
        }
    }
}

Backward: Use → Definitions

for _, b := range fn.Blocks {
    for _, instr := range b.Instrs {
        for _, op := range instr.Operands(nil) {
            if *op != nil {
                fmt.Printf("%s depends on %s\n", instr, *op)
            }
        }
    }
}

The whole picture: Referrers() goes forward (value → instructions that use it). Operands() goes backward (instruction → values it depends on).

Building SSA in Go

Go provides SSA via golang.org/x/tools/go/ssa. Here's a complete program that parses Go source, builds SSA, and prints it: 

package main

import (
    "fmt"
    "go/ast"
    "go/parser"
    "go/token"
    "go/types"
    "os"

    "golang.org/x/tools/go/ssa"
)

func main() {
    src := `package example

func Sum(n int) int {
    s := 0
    for i := 0; i < n; i++ {
        s = s + i
    }
    return s
}
`
    // 1. Parse
    fset := token.NewFileSet()
    f, _ := parser.ParseFile(fset, "example.go", src, 0)

    // 2. Type-check
    conf := &types.Config{}
    info := &types.Info{
        Types: make(map[ast.Expr]types.TypeAndValue),
        Defs:  make(map[*ast.Ident]types.Object),
        Uses:  make(map[*ast.Ident]types.Object),
    }
    pkg, _ := conf.Check("example", fset, []*ast.File{f}, info)

    // 3. Build SSA
    prog := ssa.NewProgram(fset, ssa.SanityCheckFunctions)
    ssaPkg := prog.CreatePackage(pkg, []*ast.File{f}, info, false)
    ssaPkg.Build()

    // 4. Print
    fn := ssaPkg.Func("Sum")
    fn.WriteTo(os.Stdout)
}

Three steps: parse, type-check, build SSA. Then you have the full graph to explore — blocks, instructions, values, and the edges between them.

There are two SSA implementations in Go. The one above (golang.org/x/tools/go/ssa) is the high-level, source-oriented one for building analysis tools. The compiler also has its own (cmd/compile/internal/ssa) — low-level and machine-oriented, for code generation. For program analysis, use the x/tools version.

Key Takeaways

  • One assignment per name: SSA eliminates mutable variables by giving every assignment a unique name, making analysis trivial
  • Values are immutable boxes: Each SSA value is defined once and never changes — you always know what's inside
  • Instruction vs Value: An instruction does something; a value produces a result. Many SSA objects (like BinOp) are both
  • Phi nodes merge branches: When control flow joins, a phi picks the right value based on which block we came from — no condition, just edge selection
  • Referrers() and Operands(): These two methods let you walk the entire SSA graph forward (def → use) and backward (use → def)
