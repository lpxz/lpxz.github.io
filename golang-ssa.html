Go SSA 101

SSA is the intermediate representation that powers compilers and static analysis tools. This post builds it up from scratch — from the core idea, to phi nodes, to walking the graph — then shows how SSA transforms dataflow analysis by killing the kill set.

The Problem: Mutable Variables Are Hard to Analyze

Consider this code: 

x := 1
x = x + 2
x = x * 3
fmt.Println(x)

A compiler or analysis tool wants to answer: "What is the value of x at line 4?" But x has been assigned three times. You have to trace through every assignment to figure out which one matters. In a big program with loops and branches, this gets hard fast.

The SSA Idea: Assign Once, Name Everything

Static Single Assignment says: give every assignment its own unique name. Rewrite the code above as: 

t0 = 1
t1 = t0 + 2
t2 = t1 * 3
print(t2)

Now every name has exactly one definition. If you see t1, you instantly know it equals t0 + 2. No ambiguity. No tracing needed.

The Rule: In SSA, every name is assigned exactly once. Need a new result? You get a new name. You never reuse an old one.

Values, Not Variables

In normal Go, you think in terms of variables — named boxes that hold data and can be overwritten: 

x := 1       // x is a box, it holds 1
x = x + 2    // same box, now holds 3

In SSA, each computation produces a value, and that value never changes: 

t0 = 1         // t0 is a value, it IS 1, forever
t1 = t0 + 2    // t1 is a value, it IS 3, forever

You can think of each SSA value as an immutable box — once you put something in, it's sealed shut. Need a new result? Get a new box.

The key consequence: if someone tells you "look at box t1", you know exactly what's inside without tracing through the program. There's only one place it was ever defined.

Instructions vs Values: Two Roles, One Object

In Go's SSA, there are two core interfaces:

  • Instruction — something that does something (a computation, a branch, a store)
  • Value — something that produces a result (a named, typed, immutable result)

Many SSA objects are both. A BinOp (binary operation like a + b) is simultaneously an instruction (it performs addition) and a value (it produces a result you can reference). One Go struct, two roles.

But not everything is both:

Produces a result (Value) No result
Does something (Instruction) BinOp, Call, Phi Store, Return, Jump
Just exists Parameter, Const —

Store writes to memory but produces no result you can name. Parameter is a function argument — it's a value, but no computation created it; it was given to you.

Phi Nodes: Merging Branches

What happens when a value depends on which branch was taken? 

func foo(x int) int {
    if x > 0 {
        x = 1
    } else {
        x = 2
    }
    return x
}

After the if/else, which value does x have? It depends on the path. In SSA, we can't just say x — every value has one definition. This is where phi nodes come in. 

b0:                        // entry
    t0 = Parameter {x}
    t1 = t0 > 0
    if t1 goto b1 else b2

b1:                        // then branch
    jump b3

b2:                        // else branch
    jump b3

b3:                        // after the if/else
    t2 = phi [b1: 1, b2: 2]
    return t2

t2 is a phi node. It means: if we arrived from b1, the value is 1; if from b2, the value is 2.

Phi vs Ternary: A ternary (x > 0 ? 1 : 2) evaluates a condition. A phi doesn't check any condition — it just looks at which block control flow came from. The condition was already evaluated earlier. The phi just merges the results after the fact.

A phi node is both an instruction and a value. As an instruction, it picks from its inputs based on the incoming edge. As a value, it produces a result (t2) that other instructions can use.

Loops: Phi Nodes Get Interesting

func sum(n int) int {
    s := 0
    for i := 0; i < n; i++ {
        s = s + i
    }
    return s
}

In SSA: 

b0:                              // entry
    t0 = Parameter {n}
    jump b1

b1:                              // loop header
    t1 = phi [b0: 0, b2: t3]        // i: starts at 0, updated from b2
    t2 = phi [b0: 0, b2: t4]        // s: starts at 0, updated from b2
    t5 = t1 < t0                     // i < n
    if t5 goto b2 else b3

b2:                              // loop body
    t4 = t2 + t1                     // s + i
    t3 = t1 + 1                      // i + 1
    jump b1

b3:                              // after loop
    return t2

Notice something odd in b1: 

t1 = phi [b0: 0, b2: t3]

t3 is defined in b2, which comes after b1 in the listing. The phi references a value that appears later in the text.

This is fine. Don't read SSA like sequential code. The phi declares a rule: "on each arrival to this block, pick the right value." On the first visit (from b0), t1 is 0. On later visits (from b2), t1 is t3 — which was computed in the previous loop iteration. By the time control flow arrives from b2, t3 already exists.

This is how SSA handles mutable loop variables without any mutation.

Walking the SSA Graph

Every SSA value has a Referrers() method that returns all instructions that use it. Every instruction has an Operands() method that returns the values it depends on. These two methods give you the entire graph.

Forward: Definition → Uses

// grab the first parameter (n)
param := fn.Params[0]

// find everything that uses n
for _, instr := range *param.Referrers() {
    fmt.Printf("  used by: %s\n", instr)
}
// Output: used by: t1 = i < n

One call to Referrers() and you have every use. You can chain it to walk multiple steps: 

func walkUses(v ssa.Value, depth int) {
    indent := strings.Repeat("  ", depth)
    for _, instr := range *v.Referrers() {
        fmt.Printf("%s%s\n", indent, instr)
        // if this instruction also produces a value, keep walking
        if next, ok := instr.(ssa.Value); ok {
            walkUses(next, depth+1)
        }
    }
}

Backward: Use → Definitions

for _, b := range fn.Blocks {
    for _, instr := range b.Instrs {
        for _, op := range instr.Operands(nil) {
            if *op != nil {
                fmt.Printf("%s depends on %s\n", instr, *op)
            }
        }
    }
}

The whole picture: Referrers() goes forward (value → instructions that use it). Operands() goes backward (instruction → values it depends on).

Forward analysis propagates facts from definitions to uses — constant propagation, taint tracking. Backward analysis traces from uses back to definitions — liveness, program slicing. But here's what makes SSA special: for many analyses, walking these chains directly is all you need. To see why, we need to understand what SSA does to the classical dataflow framework.

The Kill Set Vanishes

In traditional (non-SSA) dataflow, the transfer function for a block looks like this: 

OUT = GEN ∪ (IN − KILL)

GEN is what the block produces. KILL is what it destroys. The kill set exists because two assignments can target the same variable: 

x = 1      // def₁ — this gets killed
x = 2      // def₂ kills def₁
use(x)     // only def₂ reaches here

In SSA, those become distinct names: 

x₁ = 1
x₂ = 2
use(x₂)    // x₂'s def is known trivially via Operands()

No two definitions share a name. No definition can overwrite another. The kill set is always empty.

The key insight: SSA kills the kill set. The transfer function collapses from OUT = GEN ∪ (IN − KILL) to just OUT = GEN ∪ IN. Facts only grow. They never shrink.

This sounds like a minor simplification. It's not. It changes everything about how fast the analysis converges.

Reaching Definitions: The Simplest Example

Reaching definitions asks: "which definitions might still be in scope at each program point?" On traditional IR, this requires tracking gen and kill sets, iterating to a fixpoint. On SSA, the implementation is almost embarrassingly simple: 

for len(worklist) > 0 {
    b := worklist.pop()

    // IN = union of all predecessors' OUT
    newIn := union(facts[pred].Out for pred in b.Preds)

    // OUT = IN ∪ GEN
    // That's it. No kill.
    newOut := newIn ∪ gen[b]

    if newOut changed {
        facts[b].Out = newOut
        worklist.add(b.Succs...)
    }
}

Compare this with the non-SSA version, which needs to compute kill sets for every block (which definitions in this block overwrite definitions from elsewhere?), then subtract them. Here, you just union and grow.

Of course, reaching definitions on SSA is almost a joke — if you want to know what reaches an instruction, just call Operands(). The definition is the operand. But the framework is a useful template for more interesting analyses, and the "no kill" property carries over to all of them.

How Fast Does It Converge?

Since facts only grow (no kill to shrink them), how many passes over the CFG does the worklist need?

Acyclic CFG: 1 pass

No loops means no back edges. Process blocks in reverse postorder, and every predecessor is handled before its successors. One pass, done.

Single loop: 2 passes

A loop introduces a back edge — an edge that goes "backward" in the reverse postorder. On the first pass, the block at the back edge target hasn't yet seen the facts from the back edge source (it was processed earlier). Pass two picks those up. 

B1 → B2 → B3
       ↑         |
       ← ← ← ← ←     (back edge)

RPO: B1, B2, B3

Pass 1: B2 processes before B3.
         B3 generates a fact. B2 doesn't see it yet.

Pass 2: B2 picks up B3's fact via the back edge. Done.

Nested loops of depth d: d + 1 passes

This is where it gets interesting. Consider a fact born in the innermost loop that needs to reach the outermost loop header. It has to cross multiple back edges, and each one costs a pass. 

B1 → B2 → B3 → B4 → B3  (inner back edge)
       ↑    |
       ← ← ←                  (outer back edge: B3 → B2)

Def born in B4.

Pass 1: B4 has the fact. Nobody else does.
Pass 2: B3 gets it via inner back edge.
         B2 is processed before B3 in RPO — misses it.
Pass 3: B2 finally picks it up from B3.

The fact bubbles outward, one layer per pass. Each back edge crossing costs one pass because in reverse postorder, the back edge target is processed before its source — so it only sees the updated facts on the next pass.

Note the direction: going from an outer block to an inner block is just forward flow along regular edges — one pass handles it. The expensive direction is inner → outer, because that's against the flow, requiring back edge crossings.

The Doors Intuition

Here's a way to think about it that makes the pass count obvious:

Each back edge is a door that only opens once per pass.

In reverse postorder, you process blocks top-down. A back edge goes bottom-up — against the processing direction. When you arrive at the target block, the source block's update from this pass hasn't happened yet. You only see it on the next pass. The door opens, the fact walks through, and the door closes until next pass. 

depth 0:  fact → done                      (1 pass)

depth 1:  fact → door → done               (2 passes)

depth 2:  fact → door → door → done        (3 passes)

depth d:  fact → door × d → done            (d+1 passes)

Compare this with non-SSA dataflow: kill sets mean a block can shrink what flows through it, so facts might need to circulate around a loop multiple times before the interplay of gen and kill stabilizes. That can take many more iterations. With SSA, pure growth means each door only needs to open once.

Skip the Blocks Entirely

Here's the real payoff. The block-level worklist we just described works, but for many analyses on SSA you don't need it at all. Instead of iterating over blocks, you follow the def-use chains directly.

Consider constant propagation. You want to know: is this value always a constant? 

// Sparse constant propagation on SSA
// No block iteration. Just follow the graph.

worklist := [all instructions]

for len(worklist) > 0 {
    inst := worklist.pop()

    // check if all operands are constant
    if allOperandsConst(inst) {
        result := fold(inst)        // evaluate at compile time
        lattice[inst] = Const(result)

        // propagate to all uses
        for _, user := range *inst.Referrers() {
            worklist.add(user)
        }
    }
}

No blocks. No IN/OUT sets. No gen/kill. You seed the worklist with instructions, evaluate them, and follow Referrers() to propagate. The SSA graph is the dataflow graph.

This is called sparse analysis, and it's the main reason SSA exists in modern compilers. Instead of propagating facts through every instruction in every block (most of which are irrelevant), you only visit the instructions that matter. Loop nesting depth becomes irrelevant — there are no block-level passes to count.

Block-level (traditional) Sparse (SSA)
Unit of work Basic block Individual instruction
Propagation IN/OUT sets, predecessors/successors Referrers() / Operands()
Kill set Must compute Doesn't exist
Loop sensitivity Depth d → d+1 passes Irrelevant
Wasted work Visits all instructions in a block Only visits relevant instructions

The same pattern works for dead code elimination (Referrers() is empty? The def is dead), copy propagation (follow the chain, replace uses), and type inference (propagate types forward through uses).

The one place that still needs care is phi nodes. A phi merges values from different predecessors, so when propagating through a phi, you need to check all incoming operands. If they all agree (all the same constant, all tainted), the phi result inherits that fact. If they disagree, you take the lattice meet. Phis are where path-sensitivity and fixpoints sneak back in — but only locally, at join points, not across entire blocks.

Building SSA in Go

Go provides SSA via golang.org/x/tools/go/ssa. Here's a complete program that parses Go source, builds SSA, and prints it: 

package main

import (
    "fmt"
    "go/ast"
    "go/parser"
    "go/token"
    "go/types"
    "os"

    "golang.org/x/tools/go/ssa"
)

func main() {
    src := `package example

func Sum(n int) int {
    s := 0
    for i := 0; i < n; i++ {
        s = s + i
    }
    return s
}
`
    // 1. Parse
    fset := token.NewFileSet()
    f, _ := parser.ParseFile(fset, "example.go", src, 0)

    // 2. Type-check
    conf := &types.Config{}
    info := &types.Info{
        Types: make(map[ast.Expr]types.TypeAndValue),
        Defs:  make(map[*ast.Ident]types.Object),
        Uses:  make(map[*ast.Ident]types.Object),
    }
    pkg, _ := conf.Check("example", fset, []*ast.File{f}, info)

    // 3. Build SSA
    prog := ssa.NewProgram(fset, ssa.SanityCheckFunctions)
    ssaPkg := prog.CreatePackage(pkg, []*ast.File{f}, info, false)
    ssaPkg.Build()

    // 4. Print
    fn := ssaPkg.Func("Sum")
    fn.WriteTo(os.Stdout)
}

Three steps: parse, type-check, build SSA. Then you have the full graph to explore — blocks, instructions, values, and the edges between them.

There are two SSA implementations in Go. The one above (golang.org/x/tools/go/ssa) is the high-level, source-oriented one for building analysis tools. The compiler also has its own (cmd/compile/internal/ssa) — low-level and machine-oriented, for code generation. For program analysis, use the x/tools version.

Key Takeaways

  • One assignment per name: SSA eliminates mutable variables by giving every assignment a unique name, making analysis trivial
  • Values are immutable boxes: Each SSA value is defined once and never changes — you always know what's inside
  • Instruction vs Value: An instruction does something; a value produces a result. Many SSA objects (like BinOp) are both
  • Phi nodes merge branches: When control flow joins, a phi picks the right value based on which block we came from — no condition, just edge selection
  • Referrers() and Operands(): These two methods let you walk the entire SSA graph forward (def → use) and backward (use → def)
  • No kill set: Every SSA name is unique, so no definition overwrites another. The transfer function simplifies to OUT = GEN ∪ IN — facts only grow
  • Convergence scales with nesting: Acyclic: 1 pass. Single loop: 2 passes. Nesting depth d: d+1 passes. Each back edge is a door that opens once per pass
  • Sparse analysis skips blocks: Follow def-use chains directly via Referrers(). No blocks, no IN/OUT sets, no loop-depth sensitivity. This is the real payoff of SSA
