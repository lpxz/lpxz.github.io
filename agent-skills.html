Agent Skills: Separating Judgment from Capability in LLM Agents

LLMs can reason, write code, and call APIs. What they can't do is remember how you want things done. Skills fix that gap — and the data shows why this distinction matters more than adding more tools.

The Problem: Every Conversation Starts from Zero

You've spent twenty minutes explaining to an LLM exactly how to write a blog post for your site — the design system, the writing philosophy, the publish workflow. It nails it. The next day, you open a new chat and start from scratch.

This isn't a memory problem. It's a judgment problem. The LLM already knows how to write HTML, call shell commands, and generate prose. What it doesn't know is your specific workflow: that you prefer insight-first writing over step-by-step tutorials, that images should use border-radius: 16px, that you publish with git push to GitHub Pages.

The standard fix is to give the LLM more tools — file access, web search, browser automation. But consider: a developer with access to every tool on earth still doesn't know how to do your job. Tools provide capability. What's missing is judgment.

The Gap: LLMs have broad capability but zero domain-specific expert judgment. Every session, you're re-teaching the same expertise from scratch.

The Key Insight: Pre-Compiled Expert Judgment

Think about how human expertise works. A senior engineer doesn't reason from first principles every time they review code. They apply learned patterns: "this looks like a race condition," "this API design will cause versioning pain." Years of judgment, compressed into instant pattern-matching.

Skills do the same thing for LLM agents. A skill is a set of instructions — written in plain markdown — that encodes how an expert thinks about a specific class of problem. Not what tools to call. Not what API to hit. How to think.

Here's a concrete example. Without a skill, you'd prompt: 

// Without a skill: re-teach everything every time
"Write a blog post about X. Use Inter font, dark theme,
indigo accent. Lead with insights not definitions. Use
analogies. Cut algorithm steps. Add a reading progress bar.
Use the template from ifds-ide.html. Put images in /images/.
Publish with git push..."
// 500+ words of instructions, every single time

With a skill, you type /write-blog and all of that judgment loads automatically. The skill doesn't give the LLM new capabilities — it could always write HTML and run git. It gives the LLM your expert judgment about how to do it well.

Core Insight: If the hard part is getting access to data, you need a tool. If the hard part is judgment, you need a skill. Skills don't add capability — they add expertise.

The Architecture: Progressive Disclosure

The clever engineering behind skills is progressive disclosure. An LLM's context window is precious real estate. Loading every skill's full instructions at startup would waste thousands of tokens on irrelevant guidance. Instead, skills load in three levels:

Level 1: Metadata → name + description only # always loaded (~20 tokens each) Level 2: Instructions → full SKILL.md body # loaded when triggered (~500-2000 tokens) Level 3: Resources → templates, scripts, examples # loaded on demand during execution

At startup, the LLM only sees a menu: "write-blog: Write Blog Post," "youtube: Download and Screenshot." Two tokens per skill. When you type /write-blog or the LLM matches your request against a skill description, the full instructions expand into context. Templates and scripts load only when the skill actually needs them.

This is the same pattern behind virtual memory: don't load what you don't need. And it means you can have dozens of skills without paying the context cost until one fires.

Each skill is just a folder with a SKILL.md file: 

---
name: write-blog
description: Write a blog post for the site
allowed-tools: [Bash, Read, Write, WebSearch]
---

# Instructions
Lead with insights, not definitions.
Every concept needs a real example.
Use the template from ifds-ide.html...

The frontmatter controls permissions and execution model. The markdown body is the expert judgment. No API to implement. No server to deploy. Just markdown that anyone can write.

Skills vs. Tools: Complementary, Not Competitive

The most common confusion: "Aren't skills just another kind of tool?" No. They sit at completely different abstraction levels, and understanding the difference is the key to building effective agents.

Dimension Skills MCP Tools Function Calling
Provides How to think about a problem What the agent can do Structured API invocation
Abstraction High (workflow orchestration) Low (individual operations) Low (single function)
Latency Zero (local context injection) Network overhead Network overhead
Created by Anyone (plain markdown) Developers (server implementation) Developers (API implementation)
Composability Skills call tools and other skills Tools are atomic Functions are atomic

The relationship is layered: skills orchestrate, tools execute. Consider booking a pickleball court through browser automation. The MCP tools provide the capability: navigate, click, fill, screenshot. But the skill provides the judgment: which URL to navigate to, what sequence of clicks to follow, how to handle CAPTCHAs, when to retry.

Without the skill, you'd need to explain the entire booking workflow every time. Without the tools, the skill would be a set of instructions nobody could execute. They're two halves of the same system. 

// The pattern that works in practice:
Skill (orchestration + judgment)
  └─ MCP Tool: navigate to booking page
  └─ MCP Tool: fill login form
  └─ MCP Tool: click available time slot
  └─ MCP Tool: screenshot for confirmation
  └─ Judgment: retry if slot taken, try next best time

The Data: What Adoption Tells Us

The Claude Code ecosystem provides a natural experiment. Skills launched alongside MCP tools and function calling, letting us observe which extension mechanism developers gravitate toward for different problem classes.

277K+ Installs: Frontend Design Skill
70% Fewer production bugs (TDD skills)
90% Test coverage with testing skills
0ms Latency (local context injection)

The pattern in the data is clear: skills dominate for judgment-heavy tasks. The most-installed skills aren't wrappers around APIs — they're design philosophies (Frontend Design), quality methodologies (TDD), and workflow orchestrations (Remotion Video). The problems where "how to think about it" matters more than "how to access it."

Meanwhile, MCP tools dominate for capability-heavy tasks: database access, cloud provider APIs, browser automation. Nobody writes a skill to teach an LLM how to read a file — that's a capability problem, not a judgment problem.

The most effective agent setups use both. Developer surveys show that teams combining skills with MCP tools report spending more time finishing work than managing the tool — the skill handles the "what should I do?" question, and the tools handle the "how do I do it?" execution.

Real-World Impact: Three Patterns

Looking at how skills are actually used in production, three patterns emerge:

Pattern 1: Content Generation with Taste. The Frontend Design skill (277K+ installs) doesn't teach Claude how to write CSS — it already knows that. It teaches Claude to make design decisions: prefer purposeful typography over decoration, use animation for meaning not flash, build consistent color systems. The skill encodes the judgment gap between "technically correct CSS" and "design that feels intentional."

Pattern 2: Multi-Step Workflow Orchestration. Booking a court, publishing a blog post, creating a video from a script. These tasks require stringing together many tool calls in the right order with the right error handling. The skill acts as a runbook: "do this, then this, if that fails try this instead." Without it, each step requires a separate prompt.

Pattern 3: Quality Methodology. TDD skills don't add testing tools — the LLM can already run pytest. They add the discipline: write the test first, watch it fail, write minimal code to pass, refactor. The 70% reduction in production bugs comes from judgment about when and how to test, not from the ability to test.

The Recurring Theme: In every high-impact pattern, the skill's value isn't capability (the LLM already has that). It's expert judgment about a specific domain, encoded as reusable instructions.

Why This Matters Beyond Claude

The skills pattern points to something deeper about how we'll build AI systems. The current race is focused on capability: bigger models, more tools, more data. But the bottleneck in practice is often judgment — knowing which approach to take, not having the ability to take it.

This maps onto established concepts in AI research. ReAct (Yao et al., 2023) interleaves reasoning with action in a generic loop. Skills provide pre-compiled reasoning strategies for specific problem classes. Where ReAct says "think, then act," skills say "here's exactly how to think about this kind of problem."

The broader implication: as LLMs become more capable, the differentiating factor shifts from "what can the model do?" to "what does the model know about how to approach this specific task?" Skills are a low-tech, high-leverage answer to that question. Plain markdown. No fine-tuning. No API servers. Just expert judgment, made reusable.

Key Takeaways

  • Judgment vs. capability: LLMs already have broad capability. Skills fill the gap by encoding domain-specific expert judgment as reusable markdown instructions.
  • Skills orchestrate, tools execute: Skills tell the agent how to think; MCP tools and function calling give it what to do. They're complementary layers, not alternatives.
  • Progressive disclosure: Skills load in three levels (metadata → instructions → resources), paying context cost only when triggered — the same insight behind virtual memory.
  • Data supports the split: High-adoption skills cluster around judgment-heavy tasks (design, methodology, workflows), while tools dominate capability-heavy tasks (APIs, databases, browsers).
  • Democratized expertise: Anyone can write a skill in plain markdown — no API implementation, no server deployment. This makes expert judgment shareable and version-controllable.
